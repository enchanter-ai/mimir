## Attack 10 — `signature.protected_header.key_id` swapped to a different key

The `key_id` in `signature.protected_header` has been replaced with `ephemeral-attacker-00000000-0000-0000-0000-000000000000` (original: `ephemeral-31edacbc-55b5-415c-b5c7-51fe6fe77dc5`). In v2, `protected_header` (including `key_id`) is part of the signed canonical form (§9.2), so this substitution invalidates the signature (`VE-008`). A Consumer **MUST** fail DID resolution for the substituted key (`VE-003`) because the attacker-controlled `key_id` does not exist in the legitimate tool's DID document. This vector exercises the structural fix for the v1 SSRF vulnerability: in v2, any `key_id` modification is self-defeating because it breaks the signature before DID resolution can be attempted. Spec §6.13, §9.2, §10.2 step 6–7, §15.18 (Replay Across Key-Rotation Forks), §15.20 (Pre-Validation SSRF via key_id Malleability).
